# 모듈 8: Agent Teams

> 배우는 기능: AGENTS.md 작성, TF 파이프라인 설계, 다관점 검증
> OM 실습: 검수팀→분석팀→메일팀 TF 파이프라인 + AGENTS.md 작성

---

## WHY

M4에서 Subagent로 독립 작업을 병렬 처리하는 법을 배웠다.
하지만 "교안 검수 → 만족도 분석 → 보고 메일"처럼 순서가 있는 파이프라인은 병렬이 아니라 **역할 분담**이 필요하다.

Agent Teams는 여러 에이전트에게 각자 역할을 주고, 정해진 순서로 협업하게 한다.
이은지 님이 코칭한 "분석 에이전트 + 메일 에이전트 분리"의 완성형이다.

**Before (단일 에이전트):**
- 하나의 Claude Code에 모든 걸 시킴 → 뒤로 갈수록 품질 저하
- 이은지 코칭: "프롬프트가 뚱뚱하면 오류 증가"

**After (Agent Teams):**
- 검수 에이전트: PPT 분석에 집중 → 고품질 검수 리포트
- 분석 에이전트: CSV 분석에 집중 → 정확한 분류
- 메일 에이전트: 메일 작성에 집중 → 적절한 톤앤매너
- 각자 전문 영역에 집중 → 전체 품질 향상

**공식 문서:** https://docs.anthropic.com/en/docs/claude-code/agent-teams

## EXPLAIN

### Agent Teams vs Subagent

| | Subagent (M4) | Agent Teams (M8) |
|---|---|---|
| 목적 | 독립 작업 병렬 처리 | 역할 분담 + 순차 협업 |
| 비유 | 비서 3명에게 각자 다른 일 동시에 | 팀원 3명이 릴레이 |
| 구조 | 모두 같은 역할 | 각자 다른 역할 |
| 예시 | PPT 3개 동시 검수 | 검수→분석→메일 순서대로 |

### AGENTS.md = 팀 에이전트의 역할과 협업 규칙

```markdown
# OM 교육운영 TF

## 에이전트 1: 검수 담당
- 역할: 교안 PPT를 검수 프로토콜로 분석
- 출력: 검수 리포트 (REJECT/CONDITIONAL/PASS)
- 전달 대상: 분석 담당

## 에이전트 2: 분석 담당
- 역할: 만족도 CSV를 분석하여 인사이트 도출
- 출력: 분석 보고서 (긍정/부정 분류 포함)
- 전달 대상: 메일 담당

## 에이전트 3: 메일 담당
- 역할: 검수+분석 결과를 바탕으로 메일 초안 작성
- 출력: 고객사용 + 강사용 메일 초안
```

### TF 파이프라인 = 릴레이 경주

```
[검수 에이전트] → 검수 리포트 → [분석 에이전트] → 분석 보고서 → [메일 에이전트] → 메일 초안
```

각 에이전트는 이전 에이전트의 출력을 입력으로 받아 자기 역할을 수행한다.

### 다관점 검증

같은 주제를 여러 관점에서 검토:
- 찬성 관점 에이전트: "이 교육 계획의 강점은?"
- 반대 관점 에이전트: "이 계획의 리스크는?"
- 수강생 관점 에이전트: "수강생 입장에서 불편한 점은?"

## QUIZ

**Q1.** Agent Teams와 Subagent의 핵심 차이는?

- A) 차이 없다
- B) Subagent는 독립 병렬, Agent Teams는 역할 분담 + 순차 협업
- C) Agent Teams가 더 빠르다
- D) Subagent가 더 정확하다

**정답: B**
> Subagent = "비서 3명에게 각자 다른 일 동시에". Agent Teams = "팀원 3명이 릴레이로 순서대로". 용도가 다릅니다.

**Q2.** AGENTS.md에 꼭 포함해야 하는 것은?

- A) 에이전트 이름만
- B) 각 에이전트의 역할, 입력/출력, 전달 대상
- C) 코드
- D) API 키

**정답: B**
> AGENTS.md는 각 에이전트가 "무엇을 하고, 무엇을 받아서, 무엇을 내놓고, 누구에게 전달하는지"를 정의합니다.

## LEADER-TIP

1. **이은지 코칭 완성**: "분석 에이전트 + 메일 에이전트 분리" → Agent Teams로 구현. 각 에이전트가 전문 영역에 집중하므로 전체 품질 향상
2. **팀 공용 AGENTS.md**: 누가 실행해도 동일한 파이프라인으로 동일한 결과
3. **점진적 확장**: 3개 에이전트로 시작 → 효과 확인 → 4번째 추가 → 반복
4. **다관점 검증 활용**: 교육 기획안을 찬성/반대/수강생 관점으로 사전 검증

## TROUBLESHOOTING

### "Agent Teams와 Subagent를 언제 써야 할지 모르겠어요"
- 독립적 작업 여러 개 → Subagent (M4)
- 순서 있는 워크플로우 → Agent Teams (M8)
- 헷갈리면: "앞 결과가 뒤에 필요한가?" → 예: Teams, 아니오: Subagent

### "AGENTS.md를 어떻게 작성하는지 모르겠어요"
- "이 워크플로우를 AGENTS.md로 만들어줘" 요청
- 기존 AGENTS.md 예시를 참고

### "파이프라인 중간에 에러가 나요"
- 어떤 에이전트에서 에러인지 확인 (1단계? 2단계?)
- 해당 에이전트의 역할 정의를 더 구체적으로 수정
- 중간 결과물을 파일로 저장하여 디버깅

### "다관점 검증이 피상적이에요"
- 각 관점에 구체적 지시 추가: "비용 관점에서 리스크", "시간 관점에서 리스크"
- Think Harder 활용: "ultrathink로 깊이 분석해줘"
