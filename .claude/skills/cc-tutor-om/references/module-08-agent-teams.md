# 모듈 8: Agent Teams

> 배우는 기능: AGENTS.md 작성, TF 파이프라인 설계, 다관점 검증
> OM 실습: 분석팀→리포트팀→전달팀(+회수팀) 교육 종료 TF 파이프라인 + AGENTS.md 작성

---

## WHY

M4에서 Subagent로 독립 작업을 병렬 처리하는 법을 배웠다.
하지만 "만족도 분석 → 결과 리포트 → 강사/담당자 전달"처럼 순서가 있는 파이프라인은 병렬이 아니라 **역할 분담**이 필요하다.

Agent Teams는 여러 에이전트에게 각자 역할을 주고, 정해진 순서로 협업하게 한다.
교육 종료 후 업무 — 분석, 리포트 생성, 메일/Slack 전달 — 를 각 전문 에이전트가 담당한다.

**Before (단일 에이전트):**
- 하나의 Claude Code에 모든 걸 시킴 → 뒤로 갈수록 품질 저하
- 분석하면서 메일도 쓰고 Slack도 보내고 → 프롬프트가 뚱뚱해짐

**After (Agent Teams):**
- 분석팀: 만족도 데이터 분석에 집중 → 정확한 분류
- 리포트팀: 결과 리포트 생성에 집중 → 깔끔한 형식
- 전달팀: 메일/Slack 발송에 집중 → 적절한 톤앤매너
- 각자 전문 영역에 집중 → 전체 품질 향상

**공식 문서:** https://docs.anthropic.com/en/docs/claude-code/agent-teams

## EXPLAIN

### Agent Teams vs Subagent

| | Subagent (M4) | Agent Teams (M8) |
|---|---|---|
| 목적 | 독립 작업 병렬 처리 | 역할 분담 + 순차 협업 |
| 비유 | 비서 3명에게 각자 다른 일 동시에 | 팀원 3명이 릴레이 |
| 구조 | 모두 같은 역할 | 각자 다른 역할 |
| 예시 | 코치 3명 일정 동시 조회 | 분석→리포트→전달 순서대로 |

### AGENTS.md = 팀 에이전트의 역할과 협업 규칙

```markdown
# OM 교육종료 TF

## 에이전트 1: 분석팀
- 역할: 학습용 만족도 Sheets 분석 (항목별 평균, 주관식 긍정/부정 분류)
- 출력: 분석 결과 → 리포트팀에 전달

## 에이전트 2: 리포트팀
- 역할: 분석 결과를 강의 결과 리포트로 생성
- 출력: reports/결과리포트.md → 전달팀에 전달

## 에이전트 3: 전달팀
- 역할: 강사에게 Gmail 임시저장 + 기업담당자에게 Gmail 임시저장 + LD에게 Slack 공유
- 출력: 메일 임시저장 완료 + Slack 메시지 전송
```

### TF 파이프라인 = 릴레이 경주

```
[분석팀] → 분석 결과 → [리포트팀] → 결과 리포트 → [전달팀] → 메일 임시저장 + Slack
```

각 에이전트는 이전 에이전트의 출력을 입력으로 받아 자기 역할을 수행한다.

### 다관점 검증

같은 파이프라인을 여러 관점에서 검토:
- OM 관점: "운영 효율이 충분한가?"
- 강사 관점: "피드백이 유용한 형태인가?"
- 기업담당자 관점: "보고가 충분한가?"

## QUIZ

**Q1.** Agent Teams와 Subagent의 핵심 차이는?

- A) 차이 없다
- B) Subagent는 독립 병렬, Agent Teams는 역할 분담 + 순차 협업
- C) Agent Teams가 더 빠르다
- D) Subagent가 더 정확하다

**정답: B**
> Subagent = "비서 3명에게 각자 다른 일 동시에" (예: 코치 3명 일정 동시 조회). Agent Teams = "팀원 3명이 릴레이로 순서대로" (예: 분석→리포트→전달). 용도가 다릅니다.

**Q2.** AGENTS.md에 꼭 포함해야 하는 것은?

- A) 에이전트 이름만
- B) 각 에이전트의 역할, 입력/출력, 전달 대상
- C) 코드
- D) API 키

**정답: B**
> AGENTS.md는 각 에이전트가 "무엇을 하고, 무엇을 받아서, 무엇을 내놓고, 누구에게 전달하는지"를 정의합니다.

## LEADER-TIP

1. **교육 종료 자동화**: "분석→리포트→전달" 파이프라인으로 교육 종료 후 업무 2시간 → 10분
2. **팀 공용 AGENTS.md**: 누가 실행해도 동일한 파이프라인으로 동일한 결과
3. **점진적 확장**: 3개 에이전트로 시작 → 회수팀 추가 → 필요에 따라 확장
4. **에이전트 빼기도 설계**: 불필요한 에이전트를 제거하는 판단력도 중요 — "GPT 안 쓴 교육이면 회수팀 불필요"

## TROUBLESHOOTING

### "Agent Teams와 Subagent를 언제 써야 할지 모르겠어요"
- 독립적 작업 여러 개 → Subagent (M4)
- 순서 있는 워크플로우 → Agent Teams (M8)
- 헷갈리면: "앞 결과가 뒤에 필요한가?" → 예: Teams, 아니오: Subagent

### "AGENTS.md를 어떻게 작성하는지 모르겠어요"
- "이 워크플로우를 AGENTS.md로 만들어줘" 요청
- 기존 AGENTS.md 예시를 참고

### "파이프라인 중간에 에러가 나요"
- 어떤 에이전트에서 에러인지 확인 (분석팀? 리포트팀? 전달팀?)
- 해당 에이전트의 역할 정의를 더 구체적으로 수정
- 중간 결과물을 파일로 저장하여 디버깅

### "다관점 검증이 피상적이에요"
- 각 관점에 구체적 지시 추가: "비용 관점에서 리스크", "시간 관점에서 리스크"
- Think Harder 활용: "ultrathink로 깊이 분석해줘"
