# 모듈 7: Skill Creator

> 배우는 기능: 도메인 지식 → 스킬 제작, SKILL.md 구조, Eval 검증, A/B 비교
> OM 실습: /만족도분석 스킬 직접 제작 — CSV→분석→분류→메일 파이프라인을 스킬로 패키징

---

## WHY

M3에서 스킬을 "사용하는 법"을 배웠다. 이제 직접 만들 차례다.
M2에서 했던 만족도 분석 — CSV 읽기 → 분석 → 긍정/부정 분류 → 메일 초안 — 이 워크플로우를 매번 5단계로 지시하지 않고 `/만족도분석` 한 마디로 실행되게 만든다.

**Before (매번 지시):**
- "CSV 읽어줘" → "분석해줘" → "분류해줘" → "메일 써줘" → "저장해줘"
- 지시 누락 시 결과 품질 차이
- 팀원에게 전달하려면 5단계를 하나하나 설명

**After (스킬 제작):**
- `/만족도분석` → 전체 파이프라인 자동 실행
- 누가 실행해도 동일 품질
- 팀원에게는 "이 스킬 설치해" 한 마디면 끝

**공식 문서:** https://docs.anthropic.com/en/docs/claude-code/skills

## EXPLAIN

### Skill Creator = 나만의 매크로 만들기

M3에서 배운 스킬의 구조를 이제 직접 작성한다.

### SKILL.md 필수 구성 요소

```markdown
---
name: 스킬 이름
description: 스킬 설명. 트리거 키워드 포함.
---

# 스킬 제목

## 목적
이 스킬이 하는 일

## 실행 절차
1. 단계 1
2. 단계 2
3. ...

## 입력
- 필요한 데이터/파일

## 출력
- 생성되는 결과물
```

### 두 가지 스킬 유형

| 유형 | 설명 | OM 예시 |
|------|------|---------|
| **Capability Uplift** | Claude Code의 능력을 높이는 스킬 | /교안검수 (검수 프로토콜 적용) |
| **Encoded Preference** | 선호/스타일을 인코딩한 스킬 | /피드백메일 (톤앤매너 자동 적용) |

### Eval = 스킬 품질 검증

스킬을 만들었으면 "진짜 잘 되는지" 검증해야 한다.
- 같은 데이터로 스킬 실행 vs 수동 실행 비교
- 결과 품질, 일관성, 소요 시간 비교
- A/B 비교: 스킬 v1 vs v2 프롬프트 튜닝

## EXECUTE

**Step 1 — 기존 스킬 구조 분석**

    .claude/skills/cc-tutor-om/SKILL.md의 구조를 분석해줘.
    스킬이 어떤 섹션으로 구성되어 있는지 설명해줘.

SKILL.md의 메타데이터, 트리거, 레퍼런스 구조를 이해한다.

**Step 2 — /만족도분석 스킬 설계**

    다음 워크플로우를 스킬로 만들 거야. 먼저 설계만 해줘:

    1. my-data/ 또는 samples/에서 만족도 CSV 파일 찾기
    2. CSV 읽고 전체 분석 (평균 만족도, 항목별 평균)
    3. 주관식 긍정/부정 분류 (원문 보존)
    4. 분석 결과를 reports/survey-analysis.csv로 저장
    5. 고객사용 + 강사용 메일 초안 생성
    6. reports/에 저장

    SKILL.md 형식으로 설계해줘.

**Step 3 — 스킬 파일 생성**

    설계한 스킬을 .claude/skills/survey-analysis/SKILL.md로 만들어줘.

실제 스킬 파일이 생성되는 것을 확인한다.

**Step 4 — 스킬 테스트**

    /만족도분석

만든 스킬이 실제로 동작하는지 확인한다. survey-sample.csv로 테스트.

**Step 5 — Eval: 품질 검증**

    M2에서 수동으로 분석한 결과와 스킬로 분석한 결과를 비교해줘.
    - 분석 항목 동일한가?
    - 분류 결과 일치하는가?
    - 메일 톤앤매너 적절한가?

**Step 6 — A/B 개선**

    스킬의 프롬프트를 개선해줘:
    - 분류 정확도를 높이려면?
    - 메일 톤앤매너를 더 자연스럽게 하려면?

    v2를 만들고 v1과 비교해줘.

## QUIZ

**Q1.** SKILL.md의 필수 구성 요소는?

- A) 이름, 설명, 실행 절차만 있으면 된다
- B) 메타데이터(name, description), 목적, 실행 절차, 입력/출력 정의
- C) 코드만 있으면 된다
- D) 없다

**정답: B**
> 스킬은 메타데이터(이름, 설명, 트리거), 목적, 실행 절차, 입출력을 명확히 정의해야 합니다. Claude Code가 이 정보를 읽고 스킬을 실행합니다.

**Q2.** Capability Uplift 스킬과 Encoded Preference 스킬의 차이는?

- A) Capability는 능력 확장, Preference는 스타일/선호 인코딩
- B) 같은 것이다
- C) Preference가 더 강력하다
- D) Capability만 만들 수 있다

**정답: A**
> Capability Uplift: Claude Code가 못 하는 것을 할 수 있게 (예: 검수 프로토콜). Encoded Preference: 특정 스타일을 자동 적용 (예: 메일 톤앤매너).

**Q3.** 스킬 품질을 어떻게 검증하는가?

- A) 만들면 끝, 검증 불필요
- B) 같은 데이터로 스킬 vs 수동 실행 결과를 비교하고, A/B로 프롬프트 개선
- C) 코드 리뷰만 하면 된다
- D) 한 번 테스트하면 끝

**정답: B**
> Eval은 "진짜 잘 되는지" 검증입니다. 같은 데이터로 스킬과 수동 결과를 비교하고, 프롬프트를 개선하는 반복 과정이 품질을 높입니다.

## TROUBLESHOOTING

### "SKILL.md를 어떻게 작성하는지 모르겠어요"
- 기존 스킬(.claude/skills/)의 SKILL.md를 참고
- "이 워크플로우를 스킬로 만들어줘"라고 요청하면 Claude Code가 자동 생성

### "스킬이 실행되지 않아요"
- 파일 위치 확인: `.claude/skills/{스킬명}/SKILL.md`
- 메타데이터의 description에 트리거 키워드가 포함되어 있는지 확인
- 파일명이 정확히 `SKILL.md`인지 확인 (대소문자 주의)

### "스킬 결과가 기대와 달라요"
- SKILL.md의 실행 절차를 더 구체적으로 작성
- "1단계: CSV 파일을 찾는다" 보다 "1단계: my-data/에서 .csv 파일을 Glob으로 검색, 없으면 samples/에서 검색"
- Eval로 결과 비교 후 프롬프트 튜닝

### "A/B 비교 방법을 모르겠어요"
- v1과 v2를 같은 데이터로 실행
- 결과를 나란히 놓고 항목별 비교
- Claude Code에 "두 결과를 비교해줘" 요청
